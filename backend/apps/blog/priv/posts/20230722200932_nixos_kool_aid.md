## Context

I've been using [Nix and NixOS](https://nixos.org) since Jun 20, 2020.

Before this, I used to distro-hop easily once or twice every few months! I'm still no expert at `Nix`, but it cured my distrohopping, and I now run it on every computing device I own in one way or another!

At first, I was attracted to how esoteric it was (like it or not, esoteric wizardry is cool). As soon as I had gotten my setup comfy, I knew that this technology was being sat on (TwT).

I don't think this post will convince anyone to move over to `Nix` or `NixOS` but I hope it will; shilling for niche tech I like is a hobby of mine after all (UwU).

## Difficulties

One of the first things people might run into when they look into `Nix` is that `Nix` refers to one of three (at least) different things:

1. Nix, the language.
2. Nix, the package manager.
3. Nix(OS), the operating system.

I'm definitely not a `Nixlang` wizard, so this post will stop short of actually talking much about the particulars of the language, though if you're interested, I hear [this guide](https://nix.dev/tutorials/nix-language) is a great place to start.

Now, my intelligence stat isn't high enough for that yet... but fear not! With some careful pragmatism, you can still leverage a lot of benefits leveraging the `Nix` tech tree in your upgrade cycle.

I'll aim to focus on the parts of `Nix` I actually use, and to do that, we'll need to ELI5 the package manager and operating system, before delving into my configuration :)

## What is Nix (Package Manager)

Very briefly, `Nix` is a functional package manager that happens to work on BSD, Linux, Mac, and Windows!

Basically, at its core, you use `Nix` to install software just like you would via [homebrew](https://brew.sh/), [winget](https://github.com/microsoft/winget-cli), or any other package manager.

> As an aside, in the last few years, a lot of other tech has been called a "package manager" like [npm](npmjs.com) and [asdf](https://asdf-vm.com).
>
> I actually feel like "package manager" refers to, like `Nix`, three different things:
>
> - `brew` is used for installing os applications (tailscale, firefox, etc).
> - `asdf` is used for installing project dependencies (database, programming language, etc).
> - `npm` is used for installing code dependencies (libraries, gems, etc.)
>
> There are fuzzy boundaries here, but they feel very different in my head, whereas `Nix` can do all of these and more... (uwu)

You can use the command `nix-env --install <package>` to install stuff! Great! Done!

_But wait, hol' up..._

Doing that throws away all the good parts of `Nix`, and in my opinion, is totally against the philosophy of the `Nix` tech tree.

### Functional Package Management

The first difference between how `Nix` positions itself, and how say `Homebrew` positions itself, is that `Nix` is a _functional_ package manager.

In traditional package managers, when you install a package like `firefox`, you get a global installation (or maybe a per-user installation) of `firefox`. This is fine, but `firefox` might depend on specific versions of other packages that need to be installed first.

What happens if you need to install multiple pieces of software that depend on different versions of the same dependency?

Well... thankfully when you're sticking to the happy path and not doing anything too esoteric, you won't run into this because we can work around it by vendoring (or doing other equally nasty things) to sidestep the issue.

It gets trickier if you want to install multiple versions of Erlang, or Python on your operating system, hence the introducion of "version managers" and other things to work around this issue.

In short, _imperative_ package management is hard! Lots of global state, mutation is dangerous, _insert rant about functional programming here._

When you install an application with `Nix`, instead of installing a package named `firefox` into your OS' global state, you're installing some automatically derived hash to the "nix store" which gets added to your `PATH`.

That way, you can have multiple versions of the same package installed in your system at once, and you can use them all freely.

Actually, its even cooler than that, because entries in `/nix/store/` effectively contain entire standard FHS which is self-contained and includes:

- The required dependencies of a specific package.
- Any configuration for said package.
- The package itself.

BOOM. No more package hell. Every package is completely isolated from everything else. Pure.

### Declarative Packagamenet Management

Now, you don't _actually_ lose any of the functional magic of `Nix` when you try installing software via `nix-env --install <package>` since `Nix` is fundamentally functional.

However, a large part of the `Nix` philosophy also involves being _declarative_.

When you install applications via any package manager (or running `nix-env`), you're still effectively mutating global state. You can add, change, or remove packages from this global state, and it can sometimes be difficult to know the state of everything thats currently installed and why.

Enter declarative package management. This comes free when you're using `NixOS`, but on standalone installations of `Nix` you can get similar workflows by utilizing "plugins" such as:

- [flakes](https://nixos.wiki/wiki/Flakes) which lets you configure various things (I talk more about this below) natively (this is still experimental so you need to opt in).
- [home-manager](https://nix-community.github.io/home-manager/) which lets you configure packages on a user-by-user basis.
- [nix-darwin](https://github.com/LnL7/nix-darwin) which lets you configure various aspects of MacOS, as well as letting you declaratively manage `casks`.

In these declarative setups, you typically edit some configuration file such as your `/etc/nixos/configuration.nix` to define what you want installed in your system, for example:

```nix
{ config, lib, pkgs, ... }:

{
  imports = [];
  environment.systemPackages = with pkgs; [ firefox steam zsh tmux ];
}
```

Now, part of installing `Nix` actually configures something called a `nix-profile` which is tied to your user. A `nix-profile` is ultimately some _composition of packages_ called a "generation".

Mutating your systems state declaratively or via `nix-env` ultimately creates a new generation which contains all of the packages you desire.

These packages are built or fetched from a cache, symlinked into the right places, and become ready for you to use.

Additionally, these "generations" are persistent. If you try updating the version of some software and it causes an issue, you can always safetly roll back to the last known good "generation".

Thanks to the functional/pure nature of `/nix/store`, is basically assured to be good.

If that isn't a superpower, I don't know what is! (UwU)

In my mind, every other benefit that you can get out of `Nix` is an emergent property of the fact that `Nix` is able to manage packages in a functional, declarative, and generational way.

## What is Nix (Operating System)

Very briefly, `NixOS` is a functional & declarative operating system based on Linux :)

This is what happens when a mad scientist or arcane wizard decides to take all of the things we described earlier and derives the entire state of their operating system configuration via `Nix`.

### ELI5

Essentially, `NixOS` lets you configure services, users, permissions, even the kernel, via editing a single text file, and doing so functionally, declaratively, and generationally!

One of the weird quirks of `NixOS` is that it doesn't follow the standard FHS specified by POSIX:

- There is no `/bin`, `/sbin`, `/lib`, etc since all packages are in the `/nix/store/`.
- Entries in `/etc/` are additionally symlinks to entries in the `/nix/store/`.

You edit your system configuration which is typically defined in `/etc/nixos/configuration.nix` and run `nixos-rebuild switch` to instruct `Nix` to build everything you specified in your configuration, update all the required symlinks, and update your generation as needed.

Running `nixos-rebuild switch` multiple times is (almost?) guaranteed to have the exact same output -- it is idempotent.

Since you install `NixOS` by writing an initial `configuration.nix` and executing the `nixos-rebuild switch` command, upgrading (or rolling back) your generation is as reliable as reinstalling your operating system!

Never again have I had to painfully debug and fix a borked ArchLinux installation (or give up and reinstall) because of a bad nVidia driver update (UwU)!!

### Modules

What's even cooler is that you configure your setup by writing code in `nix-lang` (which I'm admittedly not an expert in). It fundamentally feels and reads like a weird hybrid of JSON and Haskell/Ocaml but it isn't too hard to jump right into.

This means your configuration is fully programmable, you can refactor complex modules to smaller modules, refer to modules from other modules, optionally enable/disable modules based on various factors, and basically do whatever you want given enough time/skill/pain/suffering.

As part of my configuration, for example, is the following module:

```nix
{ pkgs, lib, config, ... }:

with lib;
{
  options.modules.proxy = {
    enable = mkOption { type = types.bool; default = false; };
    openFirewall = mkOption { type = types.bool; default = false; };
    firewallPorts = mkOption { type = types.listOf types.port; default = [ 80 443 ]; };
    proxies = mkOption { type = types.attrsOf types.port; default = { }; };
  };

  config = mkIf config.modules.proxy.enable {
    networking.firewall.allowedTCPPorts = mkIf config.modules.proxy.openFirewall config.modules.proxy.firewallPorts;

    security.acme.acceptTerms = true;
    security.acme.defaults.email = "contact@vereis.com"

    services.nginx = {
      enable = true;
      virtualHosts =
        builtins.mapAttrs (host: port: {
          enableACME = true;
          forceSSL = true;
          locations."/".proxyPass = "http://127.0.0.1:${toString port}/";
        }) config.modules.proxy.proxies;
    };
  };
}
```

This module can be used like so in my actual operating system configuration:

```nix
  imports = [
    modules/services/tailscale.nix
    modules/services/proxy.nix
  ]

  modules.tailscale.enable = true;
  modules.proxy = {
    enable = true;
    openFirewall = true;
    proxies = {
      "sonarr.vereis.com" = 8989;
      "radarr.vereis.com" = 7878;
      "prowlarr.vereis.com" = 9696;
      "transmission.vereis.com" = 9091;
      "lidarr.vereis.com" = 8686;
      "readarr.vereis.com" = 8787;
      "printer.vereis.com" = 631;
    };
  };
```

And BOOM! When used, my module will automatically ensure `nginx` is installed, is automatically started and monitored on boot via `SystemD`, and a configuration file is automatically created based on the provided `proxies` map to automatically setup my `nginx` virtual hosts.

As a user, you don't actually care about any of the underlying complexity, you just get to easily proxy stuff :)

Of course, because I have multiple machines (and I manage my family's machines too), my configuration is structured in such a way that each machine can import and configure only modules it needs.

## Flakes

I didn't really use `flake`s until relatively recently, and you definitely don't _need_ to, but I strongly suggest that if you're starting off with `Nix` from scratch, you're better off doing so.

In my mind, `flake`s replace a lot of the manual work I needed to do in micromanaging my configuration for multiple machines, as well as in my projects. They also "fix" one of the only imperative, annoying parts of `Nix` which I haven't touched upon called `channels`.

### Replacing channels

Essentially, when you update your `NixOS` configuration (or whatever else), you're telling `Nix` that you want to install `firefox`. However, the specific version of `firefox` is tied to the state of the [nixpkgs](https://github.com/NixOS/nixpkgs) at the time of installation, or your last configured state.

Additionally, you can manage alternative package repositories which is done by "plugins" such as `home-manager` and `nix-darwin`.

This "pinning" of package repository is effectively what a `nix-channel` is. Upon installation, you get a default `nix-channel` pointing to the most recent state of `nixpkgs`. If you want to update stuff you might have to update your `nixpkgs` channel in order to be able to pull in more recent versions of your applications.

As I understand it, `nix-channels` are handled very imperatively, and thus, I lied earlier! Your `NixOS` configurations are only reproducible assuming that your `nix-channel` for `nixpkgs` is the same both times you run `nixos-rebuild switch`.

This is the wart that `flake`s solve. You opt into `flake`s by defining a `flake.nix` in the root of your projects (or as an alternative to your system configuration), and invoking this `flake` for the first time will create a `flake.lock` file which pins the version of every package you installed.

This means that you can upgrade individual packages without mutating the global state of your system. More importantly, it also means if you copy a `flake.nix` and `flake.lock` from one machine to another, you're guaranteed the exact same behaviour on both machines!

### Replacing shells

I didn't touch on this for brevity, but installing `Nix` gives you the ability to enter a `nix-shell` out of the box.

You can think of `nix-shell`s being transient `nix-profile`s that are shortlived, coupled to a single terminal, and are thrown away when you exit them.

You can create `nix-shell`s on the fly via `nix-shell -p <packages>`, or more importantly, you can add a `shell.nix` to any directory, and upon invoking the `nix-shell` command, you can automatically enter a shell containing all of the specified packages and configuration.

Because this is super convenient, I got into the habit of adding a `shell.nix` to all of my projects which automatically installs all dependencies of said project, as well as setting any environment variables as needed.

However, `shell.nix` files are relatively limited in what they can do when compared to `flake`s. Additionally, `nix-shell`s need to be built every time you enter them, and for particularly complex shells, this can take a lot of time.

There are plugins such as [lorri](https://github.com/nix-community/lorri) which help cache your `nix-shell`s, and automatically enters a `nix-shell` when you enter a directory, but this is all extra stuff that needs to be configured.

Using `flake`s, you can simply put a `flake.nix` in your project and enter it with `nix develop`. These are cached and super fast, anything you install is pinned in your `flake.lock` file, and you get all of the old benefits of `nix-shell`s plus more!

For example, this blog's `flake.nix` is pretty small, but installs `sqlit`, `Elixir` and `NodeJS`, and a few dev dependencies:

```nix
{
  inputs = {
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
    flake-utils.url = "github:numtide/flake-utils";
  };

  outputs = { self, nixpkgs, flake-utils }:
    flake-utils.lib.eachDefaultSystem (
      system:
        let
          pkgs = import nixpkgs { inherit system; };

          # `sha256` can be programmatically obtained via running the following:
          # `nix-prefetch-url --unpack https://github.com/elixir-lang/elixir/archive/v${version}.tar.gz`
          elixir_1_15_7 = (pkgs.beam.packagesWith pkgs.erlangR26).elixir_1_15.override {
            version = "1.15.7";
            sha256 = "0yfp16fm8v0796f1rf1m2r0m2nmgj3qr7478483yp1x5rk4xjrz8";
          };
        in
        with pkgs; {
          devShells.default = mkShell {
            buildInputs = [
              # API Deps ------------------------------------
              elixir_1_15_7 sqlite

              # Web Deps ------------------------------------
              nodePackages.prettier nodejs_20
            ]
              ++ lib.optionals stdenv.isLinux  ([ libnotify inotify-tools ])
              ++ lib.optionals stdenv.isDarwin ([ terminal-notifier
                                                  darwin.apple_sdk.frameworks.CoreFoundation
                                                  darwin.apple_sdk.frameworks.CoreServices
                                               ]);

            env = {
              ERL_AFLAGS = "-kernel shell_history enabled";
            };
          };
        }
    );
}
```

Projects that define a `flake.nix` can be executed without installing them manually via [nix run](https://nixos.org/manual/nix/stable/command-ref/new-cli/nix3-run.html), and more!

If this isn't a superpower, then again, I don't know what is! (UwU)

If you want to automatically enter the environment created by running `nix develop`, you can also do that via [direnv](https://direnv.net/) like so:

```sh
#!/usr/bin/env bash

if ! has nix_direnv_version || ! nix_direnv_version 2.4.0; then
  source_url "https://raw.githubusercontent.com/nix-community/nix-direnv/2.4.0/direnvrc" "sha256-XQzUAvL6pysIJnRJyR7uVpmUSZfc7LSgWQwq/4mBr1U="
fi

use flake
```

In short, `flake`s extend the wizardry that standard `Nix` gives you to stupid arcane levels. Despite their experimental status at the time of writing I can't recommend switching over to them immediately.

## My Configuration

I store my [Nix configuration](https://github.com/vereis/nixos) on GitHub.

### Installation & Top Level

Whenever I get a new machine, regardless of what it runs, I'll configure it to get access to a command line (so on Windows, I setup `WSL2` for example) and install `Nix`.

If I'm installing `NixOS`, then I have everything I need. If I'm installing `Nix` on another operating system, I recommended using the [Determinate Systems Nix Installer](https://github.com/DeterminateSystems/nix-installer), especially on MacOS.

Once that's done, I'll temporarily install `git` via `nix-shell -p git` in order to clone my repo, and from there, if my desired configuration already exists, I only need to activate it via `nixos-rebuild switch`.

This is possible because in a `flake` setup, you're able to define various different `nixosConfigurations` which can be activated depending on your `hostname`.

My top level `flake.nix` is responsible for grabbing any third party modules I want to use via `inputs`, and builds various `nixosConfigurations` or `darwinConfigurations` (specifcally for MacOS).

```nix
{
  description = "Vereis' NixOS configuration";

  inputs =
    {
      nixpkgs.url = "github:nixos/nixpkgs/nixos-unstable";
      zjstatus.url = "github:dj95/zjstatus";
      nix-darwin = {
        url = "github:LnL7/nix-darwin";
        inputs.nixpkgs.follows = "nixpkgs";
      };
      nix-homebrew = {
        url = "github:zhaofengli-wip/nix-homebrew";
      };
      home-manager = {
        url = "github:nix-community/home-manager";
        inputs.nixpkgs.follows = "nixpkgs";
      };
      nixos-wsl = {
        url = "github:nix-community/NixOS-WSL";
        inputs.nixpkgs.follows = "nixpkgs";
      };
    };

  outputs = inputs @ { self, nixpkgs, nixos-wsl, zjstatus, home-manager, ... }:
    let
      username = "vereis";
    in
    {
      darwinConfigurations = (
        import ./machines/darwin { inherit (nixpkgs) lib; inherit inputs nixpkgs home-manager nix-darwin, nix-homebrew; }
      )
      nixosConfigurations = (
        import ./machines/linux { inherit (nixpkgs) lib; inherit inputs nixpkgs home-manager nixos-wsl username zjstatus; }
      );
    };
}
```

You'll note that `darwinConfigurations` and `nixosConfigurations` simply delegate to `machines/darwin` and `machines/linux`, which are directories. If you don't manually specify a `*.nix` file to import, then a `default.nix` is used instead.

### Machine Definitions

I'll be focusing on my `nixosConfigurations` rather than `darwinConfigurations`, simply because I imagine more of you are going to be on Linux or Windows than MacOS. The configurations are basically the same so I won't lose any sleep over that!

I split up my configuration based on machine/hostname like so:

```nix
{ lib, inputs, nixpkgs, nix-darwin, home-manager, nixos-wsl, zjstatus, username, ... }:

let
  system = "x86_64-linux";
  pkgs = import nixpkgs { inherit system; config.allowUnfree = true; };
  lib = nixpkgs.lib;
in
{
  # Workstation PC
  madoka = lib.nixosSystem {
    inherit system;
    specialArgs = { inherit inputs username zjstatus; };
    modules = [
      ./madoka
      ./configuration.nix
      nixos-wsl.nixosModules.wsl
      home-manager.nixosModules.home-manager {
        home-manager.useGlobalPkgs = true;
        home-manager.useUserPackages = true;
        home-manager.extraSpecialArgs = { inherit username zjstatus; };
        home-manager.users.${username}.imports = [(import ./home.nix)] ++ [(import ./madoka/home.nix)];
      }
    ];
  };

  # Dell XPS 13
  homura = lib.nixosSystem {
    inherit system;
    specialArgs = { inherit inputs username zjstatus; };
    modules = [
      ./homura
      ./configuration.nix
      home-manager.nixosModules.home-manager {
        home-manager.useGlobalPkgs = true;
        home-manager.useUserPackages = true;
        home-manager.extraSpecialArgs = { inherit username zjstatus; };
        home-manager.users.${username}.imports = [(import ./home.nix)] ++ [(import ./homura/home.nix)];
      }
    ];
  };

  # Server / Homelab
  kyubey = lib.nixosSystem {
    inherit system;
    specialArgs = { inherit inputs username zjstatus; };
    modules = [
      ./kyubey
      ./configuration.nix
      home-manager.nixosModules.home-manager {
        home-manager.useGlobalPkgs = true;
        home-manager.useUserPackages = true;
        home-manager.extraSpecialArgs = { inherit username zjstatus; };
        home-manager.users.${username}.imports = [(import ./home.nix)] ++ [(import ./kyubey/home.nix)];
      }
    ];
  };

  ...
}
```

Each system first inherits verbatim from a top level `machines/linux/configuration.nix` that configures packages/services that I want shared between all machines (i.e. I'll explicitly install some tools such as `killall`, `lsof`; create my user; opt into `flake`s, etc.). Additionally, each system also inherits from a top level `machines/linux/home.nix` that just enables and configures `home-manager`.

Now, that the base configuration is done, each system then is configured via a `$HOSTNAME/home.nix`, a `$HOSTNAME/services.nix`, a `$HOSTNAME/default.nix`, and an automatically generated `$HOSTNAME/hardware-configuration.nix`.

### Home Modules

I try to manage everything via `home-manager` as it is able to install anything that you can install anything in `nixpkgs`, as well as specify configuration for those packages.

Now, just because something _can be installed_ doesn't mean that it can be used. You can install the `docker` binary via `home-manager`, but you don't get the service definitions and other stuff that actually makes it possible to run containers.

That means that every machine's `home.nix` is responsible for installing software I need to use as a user such as `neovim`, `steam`, `firefox`, etc. This is also responsible for any configuration each application needs.

Since these "home modules" are typically shared between different systems (not all systems opt into the same set of applications, but `neovim`, for example, has the same configuration on all systems that opt into it), I store the actual modules in `modules/home/$MODULE.nix`.

For example, my `neovim` configuration is as follows:

```nix
{ config, lib, pkgs, ... }:

with lib;
{
  options.modules.neovim = {
    enable = mkOption { type = types.bool; default = false; };
  };

  config = mkIf config.modules.neovim.enable {
    home.packages = with pkgs; [
      stylua
      sumneko-lua-language-server
      shellcheck
      shfmt
      vale
      deno
      nodePackages.prettier
    ];

    programs.fzf.enable = true;
    programs.fzf.enableZshIntegration = true;

    home.sessionVariables = {
      FZF_DEFAULT_COMMAND = "rg --files | sort -u";
      EDITOR = "nvim";
      VISUAL = "nvim";
    };

    home.file.".vale.ini".source = ./neovim/.vale.ini;
    home.file.".config/nvim/lua/config.lua".source = ./neovim/config.lua;
    home.file.".local/share/nvim/site/pack/packer/start/packer.nvim" = {
      source = builtins.fetchGit {
        url = "https://github.com/wbthomason/packer.nvim";
        ref = "master";
        rev = "afab89594f4f702dc3368769c95b782dbdaeaf0a";
      };
    };

    programs.neovim = {
      enable = true;

      viAlias = true;
      vimAlias = true;
      vimdiffAlias = true;

      withNodeJs = true;
      withPython3 = true;

      extraConfig = ''
      lua require('config')
      '';
    };
  };
}
```

When a machine configuration imports `modules/home/vim.nix`, and sets `modules.vim.enabled = true` in their configuration, my `vim.nix` module will:

- install all required packages
- setups up shell integration for all installed packages, and configures them as needed
- copies over any dotfiles into the required places

### Service Modules

For things that can't be installed via `home-manager` because they require some OS level config rather than user level config, I'll create the exact same types of modules in `modules/services/$MODULE.nix` instead and keep to a similar API as "home modules".

Because services tend to be more machine specific however, they usually allow you to configure them more. A good example of this is the `proxy.nix` modules I shared above:

```nix
{ pkgs, lib, config, ... }:

with lib;
{
  options.modules.proxy = {
    enable = mkOption { type = types.bool; default = false; };
    openFirewall = mkOption { type = types.bool; default = false; };
    firewallPorts = mkOption { type = types.listOf types.port; default = [ 80 443 ]; };
    proxies = mkOption { type = types.attrsOf types.port; default = { }; };
  };

  config = mkIf config.modules.proxy.enable {
    networking.firewall.allowedTCPPorts = mkIf config.modules.proxy.openFirewall config.modules.proxy.firewallPorts;

    security.acme.acceptTerms = true;
    security.acme.defaults.email = "contact@vereis.com"

    services.nginx = {
      enable = true;
      virtualHosts =
        builtins.mapAttrs (host: port: {
          enableACME = true;
          forceSSL = true;
          locations."/".proxyPass = "http://127.0.0.1:${toString port}/";
        }) config.modules.proxy.proxies;
    };
  };
}
```

In my actual usage, service modules tend to be minimally used. I effectively have the `proxy.nix` module, followed by modules that enable [Tailscale](https://tailscale.com), torrent clients, printer support, etc.

## What's next?

While I can definitely appreciate how daunting the `Nix` tech tree is, I hope this post presents it in a way that at very least makes sense!

It can definitely feel like a parallel path compared to people using more standard tools like `docker`, `ansible`, etc to achieve similar results, but the killing feature for `Nix` in my opinion is that you only need to install one tool.

Once you learn `Nix`, and you get it configured, you can use it to manage _everything_ in your operating system, your projects, your users, etc. You can even build minimal and pure [docker containers](https://nix.dev/tutorials/nixos/building-and-running-docker-images.html) without installing `docker`!

I'm definitely the only weirdo at [Vetspire](https://vetspire.com) that uses `Nix`, and was one of a very small handful at [Erlang Solutions](https://erlang-solutions.org), but I can definitely say its given me superpowers that I don't think I can live without!
